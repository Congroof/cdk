# 增加公告管理与公开获取

## 目标

为后续客户端接入提供公告读取能力，并在现有管理页面中提供公告创建和修改入口。

## 初始需求

- 提供无需登录即可获取公告的客户端接口。
- 提供需要管理员 JWT 鉴权的公告创建/编辑接口。
- 在管理端页面增加公告编辑入口。
- 已有公告可在管理端加载并修改。
- 公告数据需要持久化，服务重启后仍可读取。

## 已确认的项目约束

- 系统支持多个后台用户，CDK、封禁和反馈数据均以当前 JWT 用户作为归属边界。
- 客户端公开接口已有 `/api/client/u/{username}/...` 模式，用用户名解析归属用户，无需客户端鉴权。
- 后台受保护接口统一嵌套在 JWT 中间件下，可从 `Claims.sub` 得到当前用户名。
- 管理页面采用 Dashboard 标签页结构，公告管理适合新增独立标签和编辑组件。
- 数据库结构需要同时更新启动时 `CREATE TABLE IF NOT EXISTS`、手动 migration 和 Docker 初始化 schema。

## 验收标准

- 管理员可在 Dashboard 的“公告管理”标签页创建自己的公告，再次保存时更新同一条记录而不是新增记录。
- 管理员只能读取和修改 JWT 当前用户所属的公告。
- 管理端 `GET /api/announcement` 和 `POST /api/announcement` 未携带有效 Token 时返回 401。
- 客户端无需 Token 即可通过 `GET /api/client/u/{username}/announcement` 读取已启用公告。
- 尚未创建公告或公告已停用时，公开接口返回 HTTP 200 和 `data: null`。
- 用户名不存在时，公开接口返回 HTTP 404。
- 标题为空、正文为空、标题超过 128 字符或正文超过 10,000 字符时，保存接口返回 HTTP 400。
- 标题和正文保存前去除首尾空白，正文内部换行原样保留。
- 页面刷新后仍能加载当前公告；启用开关、标题、正文和更新时间展示正确。
- API 文档包含管理端及客户端接口契约和示例。

## 已确认产品决策

- 每个管理员独立维护一份公告。
- 管理端读写根据 JWT 当前用户隔离，不能查看或修改其他管理员的公告。
- 客户端通过 `/api/client/u/{username}/announcement` 免鉴权读取指定管理员的公告。
- 每个管理员只维护一条当前公告，不提供历史记录或多条轮播。
- 公告字段为 `title`、`content`、`is_enabled` 和服务端维护的 `updated_at`。
- 公开接口在尚未创建公告或公告已停用时返回 HTTP 200、`success: true`、`data: null`。
- 公开接口中的用户名不存在时返回 HTTP 404。
- 管理端提供受 JWT 保护的 `GET /api/announcement`，读取当前管理员的公告草稿（包括停用状态）。
- 管理端提供受 JWT 保护的 `POST /api/announcement`，以 upsert 方式同时处理首次创建和后续修改。
- 管理页面新增独立“公告管理”标签页；无数据时呈现创建状态，有数据时呈现修改状态，共用同一编辑表单。
- 公告标题和正文均为纯文本，不解析 HTML 或 Markdown；标题最长 128 字符，正文最长 10,000 字符。

## 范围外

- 多条公告、公告历史、轮播和删除能力。
- 定时发布、过期时间和已读状态。
- HTML/Markdown 富文本渲染。
- 不带用户名的全局客户端公告接口。
