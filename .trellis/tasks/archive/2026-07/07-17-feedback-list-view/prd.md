# 优化反馈列表查看体验

## 目标

改善管理端“用户反馈”列表的可读性与整齐度。列表只承担快速浏览和状态判断，完整反馈与管理员回复通过“查看反馈”入口集中展示，避免回复内容撑高单行并造成各行高度不一致。

## 已确认事实

- 当前界面由 `frontend/src/components/FeedbackList.tsx` 实现。
- 当前管理员回复直接渲染在“反馈内容”单元格内；回复长度会直接增加表格行高。
- 列表已经提供“回复 / 编辑回复”和“标记完成 / 重新打开”操作。
- 项目已有统一的深色弹窗样式，不需要引入新的 UI 依赖。
- 本次体验问题是前端展示问题，不需要改变反馈 API、数据模型或数据库。

## 需求

- 管理员回复不再直接完整展示在列表行内。
- 每条反馈提供清晰的“查看反馈”按钮。
- 点击按钮后以弹窗展示该条反馈的完整信息，至少包括完整反馈内容、基础元数据、关联信息、状态、提交时间以及管理员回复和回复时间。
- 无管理员回复时，详情中提供明确的未回复状态，不留出突兀的空白区域。
- 列表中的反馈内容使用受控的摘要展示，避免长反馈正文继续造成明显的行高差异。
- 保留现有回复/编辑回复、完成/重新打开能力及其接口行为。
- “回复 / 编辑回复”继续保留为列表中的独立快捷按钮，不要求管理员先打开详情。
- 新弹窗沿用项目现有 Tailwind 深色视觉规范，并支持点击遮罩或关闭按钮退出。

## 验收标准

- 有回复和无回复的反馈在列表中保持一致、紧凑的视觉节奏，不再因回复正文导致行高显著不同。
- 每一行都能通过“查看反馈”打开对应详情，不会串行或显示错误数据。
- 详情弹窗可以完整查看长反馈和长回复，文本能正确换行。
- 详情弹窗正确显示空联系方式、空机器码、空 CDK、空回复等情况。
- 现有筛选、搜索、刷新、分页、回复保存和状态切换行为不回归。
- 前端 lint 与构建通过。

## 范围外

- 修改后端 API 或数据库结构。
- 新增反馈字段或改变回复与完成状态的业务规则。
- 重做整个反馈页面的信息架构、筛选器或统计卡片。

## 已确认的产品决定

- 列表新增“查看反馈”按钮，同时保留“回复 / 编辑回复”快捷按钮。
