# SkinForge OSS 更新分发接口

## Goal

让 cdk-server 只负责 SkinForge 软件版本和 Hash 字典的元数据管理、云文档文件标识持久化及临时 OSS 直链刷新，不再通过 Nginx 直接承载安装包、Hash TXT 或 gzip 大文件。

## Confirmed Facts

- 云文档 `download_url` 带 `Expires`/`Signature`，属于临时 OSS 直链。
- 稳定可持久化的云文档标识是 `file_id` 和 `link_id`；服务端可据此重新换取下载 URL。
- cdk-server 当前定时下载 CommunityDragon 的 `hashes.game.txt`，本地生成 gzip，并由 Nginx 静态分发。
- cdk-server 已有 JWT 管理端 API、React 管理后台、MySQL schema/migration 模式。
- SkinForge 当前只发布 Windows x86_64 NSIS updater 产物。
- Tauri v2 动态更新服务在无更新时返回 HTTP 204；有更新时返回包含 `version`、`url`、`signature`，以及可选 `notes`、`pub_date` 的 JSON。

## Requirements

- 增加可复用的云文档客户端，支持上传文件、保存 `file_id/link_id`，以及按需刷新 OSS 下载 URL。
- 管理后台提供云文档配置录入和更新界面；完整 Cookie、`group_id`、`parent_id`
  等配置持久化到数据库，服务重启后继续生效。
- 软件安装包和 Hash TXT/gzip 共用同一个 `group_id + parent_id` 目录，不分别
  配置 release/hash 目录。
- Cookie 中的 CSRF 由服务端解析，不要求用户重复录入同一凭证字段。
- 保存新云文档配置前必须验证凭证和目录可用；验证失败时不得覆盖旧配置。
- Cookie 使用 AES-256-GCM 加密存储；数据库保存密文和随机 nonce，主密钥从服务器
  环境变量读取。主密钥缺失或无法解密时，上传和 OSS 换链必须失败并返回明确错误。
- 管理端配置查询不返回 Cookie 原文，只返回是否已配置、脱敏摘要和更新时间；更新
  Cookie 时必须重新提交完整值。
- 所有通过 JWT 认证的后台用户都可以查看脱敏配置状态并更新云文档配置；配置记录
  保存最后修改用户和时间。
- 增加 SkinForge 发布信息持久化，至少包含版本、平台/架构、更新说明、发布时间、Tauri 签名、安装包文件标识、文件名、大小和摘要。
- 软件发布固定为 Windows x86_64 NSIS 产物，MVP 不设计多平台发布集合。
- 软件发布表只保存一条当前记录；管理后台导入并校验新发布后覆盖旧记录。
- 软件发布和 Hash 字典均为全局单例资源，不设置 `created_by`，不按用户、CDK 或
  租户隔离。
- 所有通过 JWT 认证的后台用户都可以导入并覆盖当前全局软件发布；采用最后一次成功
  提交生效。
- 增加受 JWT 保护的发布管理 API，并与现有管理后台集成。
- 管理后台支持导入 SkinForge 构建流程生成的发布 JSON，展示校验结果，并在管理员
  确认后保存为当前发布；MVP 不由构建脚本直接调用发布 API。
- 导入预览界面允许填写或修改更新说明，确认发布前要求非空；保存后作为 Tauri
  动态响应的 `notes`。
- 导入版本必须是合法 SemVer；已有当前记录时，新版本必须严格更高，同版本覆盖和
  降级均拒绝。
- 管理后台通过本地文件选择器读取 `.json` 发布清单，展示版本、文件名、大小、
  SHA-256、发布时间、`file_id/link_id` 和签名信息；不上传安装包主体。
- SkinForge 侧构建与上传解耦，后台只消费 `release:upload` 生成的完整发布 JSON，
  不接收安装包文件主体。
- SkinForge 构建机使用自己的 `KDOCS_COOKIE` 环境变量上传安装包；cdk-server 不提供
  读取或导出数据库 Cookie 原文的接口。
- 增加公开的 Tauri 动态更新 API，根据客户端当前版本/平台/架构返回 204 或 Tauri 所需 JSON；返回的 `url` 是请求时换取的 OSS 临时直链。
- Hash 同步成功后生成 gzip，同时将规范 TXT 和 gzip 上传到云文档，持久化两个
  产物各自的文件标识、大小和摘要。
- TXT 和 gzip 采用成对原子发布：两个文件上传成功并完成 OSS 换链验证后，才更新
  当前公开 Hash 记录；任一步失败继续公开上一版本。
- 发布新软件或新 Hash 后不自动删除旧云文档文件；数据库只保留当前文件标识。
- 保留启动时和周期性 Hash 检查，并增加受 JWT 保护的手动同步 API 及后台按钮。
- Hash 上游 URL 和自动同步周期继续来自服务器环境变量，默认使用 CommunityDragon
  和 24 小时间隔；后台不提供修改入口。
- 管理后台展示当前 Hash 版本、TXT/gzip 云文档上传状态、最后成功时间、当前运行状态
  和最近错误。
- 增加公开 Hash 元数据 API，返回规范 TXT 的版本/大小/SHA-256，以及请求时换取的 OSS 下载 URL。
- Tauri updater 和 Hash metadata API 均不要求 CDK 或 JWT；管理端配置、发布和手动
  同步接口继续要求 JWT。
- API 不代理大文件内容；云文档换链失败时返回明确的服务端错误，不回退到 Nginx 大文件传输。
- 元数据 API 继续通过当前 HTTP/80 服务提供；返回的大文件 OSS URL 使用云文档的
  HTTPS 地址。
- OSS 临时 URL 使用进程内短期缓存；从 URL `Expires` 参数读取有效期，距离过期不足
  5 分钟时刷新。
- 上传或数据库提交失败不得发布半完成版本；旧的软件发布和旧 Hash 产物继续可用。

## Acceptance Criteria

- [ ] 任一已登录用户可以录入一个严格递增的 Windows x86_64 SkinForge 发布版本。
- [ ] Tauri 更新 API 对旧版本返回 200 和有效动态更新 JSON，对最新版本返回 204。
- [ ] Tauri 响应中的安装包地址来自云文档 OSS，且签名是 `.sig` 文件内容。
- [ ] Hash 同步生成的云文档产物可由公开元数据 API 获取临时 OSS URL。
- [ ] TXT/gzip 任一上传或换链失败时，公开 Hash 版本保持不变，不出现跨版本组合。
- [ ] 后台可以手动触发 Hash 同步；同步运行中不会并发启动第二次任务，并能查看状态。
- [ ] OSS URL 过期后，仅通过 `file_id/link_id` 刷新即可恢复下载，不需重新上传。
- [ ] Nginx 不再提供安装包、Hash TXT 或 gzip 的主体下载。
- [ ] 关键发布校验、版本比较、Hash 上传状态和云文档响应解析有自动化测试。
- [ ] 后端、前端、数据库初始化、迁移和部署文档同步更新。

## Out of Scope

- 自建 OSS/CDN。
- 由 cdk-server 代理或缓存大文件响应体。
- 更换 Tauri updater 签名密钥。
- 多渠道灰度发布和自动回滚到旧软件版本。
- macOS、Linux、Windows 非 x86_64 架构的软件发布。
- 软件发布历史、历史版本列表和旧版本重新启用。
- 软件更新暂停/启用开关。
- 云文档旧文件自动清理、保留数量和保留周期策略。
- HTTPS 域名、证书、443 端口和强制 HTTPS 跳转。

## Open Questions

- 无阻塞问题。

## Decisions

- 软件发布使用管理后台导入模式：构建/upload 流程输出完整发布 JSON，管理员在后台
  预览、校验并确认发布。
- MVP 不引入构建机 API Token，也不支持构建完成后自动发布。
- cdk-server 管理后台不承担安装包上传；大文件由 SkinForge 构建机直接上传云文档。
- 数据库中的云文档 Cookie 只供服务端 Hash 上传和 OSS 换链使用，不作为构建机凭证
  分发渠道。
- 构建机上传记录中的 `file_id/link_id` 必须能由服务端配置的云文档账号换取下载
  URL；管理后台导入时通过换链探测验证这一点。
- 更新说明由管理后台维护，不依赖 SkinForge PowerShell 脚本中的硬编码文本。
- JSON 解析、字段约束或 OSS 换链探测任一失败时，不得覆盖当前软件发布。
- 尚无软件发布记录时允许导入任意合法 SemVer；版本比较在服务端再次执行，不能只
  依赖浏览器预览校验。
- 当前软件记录没有 enabled/paused 状态；存在记录且请求版本较低时直接提供更新。
- 公开 API 只能返回发布/Hash 元数据和 OSS 临时地址，不得返回 Cookie、CSRF、
  数据库配置详情或后台审计信息。
- 数据库不持久化临时 URL。未接近过期的内存缓存可在换链接口短暂故障时继续使用；
  缓存缺失或进入刷新窗口后换链失败则返回服务不可用错误。
- 本次部署保持现有 HTTP API 入口，不修改 TLS 架构。
- Hash 同时上传规范 TXT 和 gzip；客户端优先使用 gzip，并保留 gzip 异常后一次
  TXT 回退。两个 OSS 产物必须对应同一规范版本和 SHA-256。
- 不实现旧 `/skinforge/latest.json` 和旧 Hash metadata URL 的兼容响应；客户端
  直接改用新 API。
- 首个切换 endpoint 的客户端通过现有外部分发方式提供；确认新客户端和新 API 后
  移除 Nginx 大文件 location。
- 只保存最新软件版本。新发布必须先通过字段校验和云文档 OSS 换链探测，再原子覆盖
  当前记录；不实现历史发布管理。
- 公开更新 API 为无用户名的全局路由；所有 SkinForge 客户端读取同一当前软件版本
  和同一 Hash 字典。
- 当前软件发布记录保存 `updated_by` 作为最后修改者审计字段，但不据此做权限限制。
- 云文档配置由管理后台维护并持久化到数据库，不再以服务启动环境变量作为主要来源。
- Cookie 加密主密钥不存数据库，由服务器环境变量长期稳定提供，以保障服务重启后
  仍可解密已有配置。
- 云文档配置权限与软件发布权限一致，不新增角色系统；任何用户和接口均不能读取
  已保存 Cookie 的原文。
- 启动同步、周期同步和手动同步共享一个进程级互斥状态；重复触发返回“正在同步”，
  不排队执行。
- 云文档数据库配置不包含 Hash 上游和调度周期；运行时调度仍由部署配置控制。
- 已验证的新 TXT/gzip 可作为本地待发布产物保留；后续同步优先补做缺失上传，避免
  在上游版本未变化时重复下载。
- 云文档旧文件由管理员人工清理；删除动作不属于发布事务，也不影响当前记录提交。
- Tauri 动态更新 API 仅为 `windows-x86_64` 解析和返回发布；其他平台/架构返回
  HTTP 204。
