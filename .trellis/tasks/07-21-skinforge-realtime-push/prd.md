# SkinForge CDK 单机换绑失效

## 目标

为 SkinForge Tauri 2 客户端增加由 `cdk-server` 主动下发的低资源实时失效通道。当一个 CDK 被换绑到新机器时，旧机器上的客户端应立即获知授权已失效，确保同一 CDK 同时只能在一台机器上使用。

## 已确认事实

- 服务端仓库为 `/Users/leibuyun/Documents/github.com/cdk-server`，后端使用 Rust、Axum 0.8、Tokio、SQLx 和 MySQL。
- 生产容器内同时运行 Nginx 与 Axum；Nginx 监听 80，Axum 默认监听容器内 `0.0.0.0:3000`。
- 当前公网客户端地址是 `http://62.234.58.74/api`，尚未使用 HTTPS。
- 本期明确继续使用 HTTP/WS，不升级 HTTPS/WSS；产品方接受明文链路可能被监听或篡改的已知风险。
- SkinForge 使用 Tauri 2；Rust 侧已有 Tokio、reqwest 和 `tokio-tungstenite`，并已有 LCU WebSocket 生命周期管理代码可参考。
- 客户端当前固定使用租户用户名 `a`，通过 `/api/client/u/{username}/...` 访问公告、激活和反馈接口。
- 公告由后台 JWT 用户管理，客户端公告读取当前是无认证 HTTP GET。
- CDK 换绑发生在 `POST /api/client/u/{username}/activate`：已激活 CDK 收到不同 `machine_code` 时，当前实现直接更新 `cdkeys.machine_code` 并返回“CDK 换绑成功”。
- 当前换绑更新没有数据库事务或旧机器码条件保护；并发换绑时需要先修正为可确定旧绑定的原子更新，再触发定向失效。
- Tauri `CdkGate` 在启动时校验，此后每 30 分钟校验一次；一次校验发生网络错误时会关闭主界面，但两次校验之间旧机器最长仍可继续运行约 30 分钟。
- 产品规则允许任何持有正确 CDK 的用户直接换绑，不增加管理员审批、一次性换绑令牌或防抢回机制；旧用户知道 CDK 时仍可再次换回，属于已接受行为。
- 现有 `usage_logs` 保存 `machine_code`、`cdk_code`、`action`、租户与时间，但激活日志在业务校验前写入且单行只能表达一个机器，不能可靠表示一次成功换绑的旧机器与新机器。
- 生产 Nginx 已设置 `X-Real-IP` 与 `X-Forwarded-For`，Axum 3000 端口未直接映射到公网；后端可在只信任该反向代理的前提下读取真实客户端 IP。
- 日使用人数峰值约 600，近两个月约 400，预计同时在线连接数不超过每日用户量。
- 生产服务器总内存 2GB，当前 `available` 约 876MB，无 Swap。

## 核心需求

1. Axum 服务端提供一个面向 SkinForge 客户端的 WebSocket 端点，并通过现有 Nginx 80 端口转发 Upgrade 请求。
2. Tauri Rust 后台维护连接，不依赖 WebView 直接持有 WebSocket；连接状态和业务事件通过 Tauri 事件系统交给前端。
3. 连接必须具备心跳、断线重连、指数退避与随机抖动，避免服务恢复时的重连风暴。
4. 服务端连接注册表和单连接发送队列必须有界；慢客户端不得无限占用内存。
5. 消息采用版本化 envelope，至少包含事件 ID、事件类型、协议版本、发生时间和业务 payload。
6. 第一阶段按单实例、最多 3000 个在线连接设计，不引入 Redis、MQTT、Kafka、RabbitMQ 或 NATS。
7. 首期只处理 CDK 换绑导致的旧机器授权失效，不建设通用公告或聊天推送。
8. 数据库中的 CDK 当前绑定关系始终是授权状态的唯一权威来源；已认证连接收到明确换绑失效事件后立即撤权，无需再等待 HTTP 校验。启动、重连和普通断线恢复仍通过数据库校验确认状态。
9. 换绑事件只应定向发送给该 CDK 的旧机器连接，不广播给其他客户端。
10. WebSocket 不得允许服务端消息直接执行任意本地命令。
11. 换绑数据库更新必须先成功提交，之后才能通知旧机器；推送失败不得回滚已经完成的新机器绑定。
12. WebSocket 失联不能让客户端无限期保持已授权状态；客户端必须在约定宽限期内恢复连接或通过 HTTP 重新校验，否则回到 CDK 门禁页。
13. 断线宽限期固定为 60 秒：客户端在此期间重连并尝试权威 HTTP 校验；60 秒内未恢复授权确认则关闭主功能并回到 CDK 门禁页。
14. 本期 WebSocket 地址使用现有公网 80 端口下的 `ws://` 路由，Nginx 同时保留现有 HTTP API；TLS、证书和 443 端口不在本期范围。
15. 换绑继续由现有客户端激活接口直接发起；不得新增管理员批准步骤或换绑冷却期。
16. 本期保存最小、追加式 CDK 绑定历史：首次激活和每次成功换绑均记录 CDK、旧机器码（首次激活为空）、新机器码、请求客户端 IP、所属租户与发生时间。
17. 绑定历史只记录数据库确认成功的激活/换绑，不把失败请求当作“使用过的机器”；失败尝试审计留待后续安全审计功能。
18. 旧机器收到换绑失效事件后必须立即把 Rust 授权状态标为无效、停止授权长连接与受保护后台循环、删除本地 `license.cdk`，并使前端回到 CDK 门禁页。
19. 换绑失效提示固定为“此 CDK 已在其他设备激活，请重新输入 CDK”。旧用户仍可手动重新输入同一 CDK并按既定规则再次换绑。
20. 授权失效采用“立即撤销、在安全点停止”：立即锁定 UI、禁止新受保护操作并取消可安全取消的后台循环；已经进入文件写入或皮肤应用原子阶段的操作允许完成当前安全步骤，随后停止，不得进入下一阶段。

## 初始资源边界

- 在线连接硬上限：3000。
- 服务端心跳间隔：30 秒；客户端无有效授权确认的最长时间为 60 秒。
- 单条 WebSocket 消息上限：64KB。
- 单客户端待发送队列目标：32 条；队列持续满时断开慢客户端。
- 默认关闭 WebSocket 消息压缩。
- 重连退避上限：60 秒，并加入随机抖动。

## 验收标准（初稿）

- Nginx 80 端口可以完成 WebSocket Upgrade，并继续兼容所有现有 HTTP API。
- 600 个空闲测试连接保持稳定时，服务端无无界任务、队列或内存增长；实际 RSS 增量被记录。
- 客户端启动后自动连接；网络中断和服务端重启后可自动恢复。
- 已授权客户端失去 WebSocket 后立即进入 60 秒宽限倒计时；倒计时内恢复连接或 HTTP 校验成功则保持可用，否则在不晚于第 60 秒回到门禁页。
- CDK 成功从旧机器换绑到新机器后，服务端只向旧机器的在线连接发送授权失效事件。
- 旧机器客户端收到已认证连接上的明确失效事件后立即进入不可用/激活门禁状态，不用等待额外 HTTP 往返。
- 旧机器收到有效失效事件后无需等待 60 秒宽限，立即删除 `license.cdk` 并显示指定提示；60 秒宽限只适用于连接异常而未收到明确失效事件的情况。
- 失效后所有新受保护命令立即被 Rust 授权状态拒绝；已运行任务不会被粗暴终止在半写状态，并能在下一安全检查点停止。
- 新机器成功换绑后保持可用，其他 CDK 和机器不受影响。
- 两个不同机器并发请求换绑时，数据库最终绑定、成功响应和被通知的旧机器顺序可确定；客户端侧可能重叠使用的时间不得超过约定的断线宽限期。
- 首次激活和成功换绑后均产生一条绑定历史，能够在数据库中还原某 CDK 使用过的机器码、换绑顺序、请求 IP 和时间；失败激活不产生成功历史。
- 非法或超限消息被拒绝，不导致服务端或 Tauri 崩溃。
- 现有公告、CDK、反馈、更新和 Hash API 回归测试继续通过。

## 暂不纳入范围

- Tauri 完全退出后的操作系统级远程推送。
- 多节点服务端之间的连接同步与消息总线。
- 通过 WebSocket 传输大型文件、Hash 内容或安装包。
- 通用聊天、客户端间通信或客户端远程控制。
- HTTP/WS 到 HTTPS/WSS 的迁移。
- 管理员审批换绑、一次性换绑令牌、换绑冷却期或阻止旧用户再次换回。
- 完整审计查询 API、筛选报表、失败尝试审计和管理界面。

## 已确定的可靠性与身份边界

- WebSocket 凭据仅由 Tauri Rust 持有。连接请求通过 Header 携带 CDK 与本机 HWID，服务端在 Upgrade 前按租户、CDK 状态、有效期和当前 `machine_code` 查询数据库；凭据不放入 URL、WebView 或前端状态。
- 服务端不持久化待投递失效消息。在线旧机器通过定向事件立即失效；离线或丢事件的旧机器依靠启动校验、重连校验和最长 60 秒租约失效。
- 连接握手失败或重连后发现机器码不再匹配时，客户端不等待剩余宽限，立即执行换绑失效流程。
- 当前链路为明文 HTTP/WS，Header 只能避免常规 URL/access log 泄漏，不能抵御网络监听；该风险已明确接受。
